# Changelog

## 2026-04-19

### Convex dev unblocked
- **Backend test typing (`packages/backend/convex/reservations.test.ts`, `packages/backend/convex/import-meta-glob.d.ts`):** removed the unresolved `vite/client` triple-slash reference from the Convex test file and replaced it with a local ambient `ImportMeta.glob()` declaration so `npx convex dev` can typecheck and start again. This restores the current backend function validators, including `tables:list({ referenceTimestamp })`, instead of leaving the web app pointed at a stale server shape.
- **Reservation availability guard (`apps/web/src/components/reserve/reservation-form-sheet.tsx`):** the reserve sheet now skips `reservations.checkAvailability` until the chosen date/time is actually in the future. This prevents the client from spamming predictable server errors when the sheet opens on an already-past slot such as `today 10:00`, while preserving the existing inline warning that asks the user to choose a future time.
- **Reservation default slot UX (`apps/web/src/components/reserve/reservation-form-sheet.tsx`):** the reserve sheet now opens on the next valid booking slot instead of always defaulting to `10:00` today. Same-day opens after business start advance to the next hour, late-night opens roll to tomorrow at `10:00`, out-of-range prefilled dates are clamped into the bookable window, and the “checking slot” state no longer appears when availability is intentionally skipped for an invalid past selection.
- **Reservation auto-refresh stability (`apps/web/src/routes/_customer.reserve.tsx`, `apps/web/src/components/reserve/reservation-form-sheet.tsx`):** the reserve sheet now keeps a stable selected-table snapshot across the 10-second floor-plan refresh, so the form is no longer treated as closed/reopened on every refetch. Combined with the table-ID-scoped guest reset, `guestCount` now stays intact while the user is editing the same table instead of silently snapping back to `1`.
- **Reservation date picker (`packages/ui/src/components/calendar.tsx`, `packages/ui/package.json`, `apps/web/src/components/reserve/reservation-form-sheet.tsx`):** added shadcn `Calendar` (react-day-picker + date-fns) to the shared UI package and replaced the native `input type="date"` on the reserve sheet with a Popover + Calendar control that respects the same min/max booking window and shows an Indonesian medium date label on the trigger.
- **Reservation start time (`packages/ui/src/components/select.tsx`, `apps/web/src/components/reserve/reservation-form-sheet.tsx`):** added shadcn `Select` (Base UI) and replaced the native `<select>` for hourly slots with the same component family, matching the rounded trigger styling and using `modal={false}` so the dropdown behaves better inside the reservation sheet.

### Pakasir QRIS migration
- **Provider migration (`packages/backend/convex/payments.ts`, `packages/backend/convex/schema.ts`, `packages/backend/convex/http.ts`):** reservation payments now use Pakasir QRIS end-to-end. Checkout creation calls Pakasir `transactioncreate/qris`, webhook handling moved to `/pakasir/webhook`, manual admin sync now verifies through `transactiondetail`, cancellation uses `transactioncancel`, and payment rows persist QR-specific metadata (`provider`, `paymentMethod`, `paymentNumber`, `fee`, `totalPayment`, `completedAt`, `expiresAt`).
- **Customer QR checkout (`apps/web/src/components/reserve/reservation-payment-sheet.tsx`, `apps/web/src/components/reserve/reservation-form-sheet.tsx`, `apps/web/src/routes/_customer.my-reservations.tsx`, `apps/web/src/routes/_customer.reserve.tsx`):** redirect-based payment was replaced with an in-app QRIS sheet showing the QR string, countdown, cancel flow, payment guide, and QR download action. Pending reservations can resume the same QRIS checkout from **My Reservations** without leaving the app.
- **Operational cleanup (`apps/web/src/routes/admin.payments.tsx`, `packages/backend/convex/emails.ts`, `packages/backend/convex/notifications.ts`, `docs/NOTES.md`):** admin copy and manual sync messaging now reference Pakasir, reservation emails/notifications read payments through the indexed reservation-payment lookup, refund notifications point to the Pakasir dashboard, and backend env docs now describe the Pakasir configuration and webhook verification model.
- **Operator handoff (`docs/NOTES.md`):** added a dedicated “Next step user actions” checklist covering Pakasir project setup, Convex env vars, webhook registration, deploy/restart, smoke-test flow, admin sync validation, and QR expiry verification.
- **Findings follow-up (`packages/backend/convex/payments.ts`, `packages/backend/convex/reservations.ts`, `packages/backend/convex/emails.ts`, `packages/backend/convex/notifications.ts`, `apps/web/src/components/reserve/reservation-payment-sheet.tsx`, `apps/web/src/routes/_customer.my-reservations.tsx`, `apps/web/src/components/reserve/reservation-form-sheet.tsx`):** checkout creation is now serialized with a short reservation lock to avoid duplicate live QR orders, pending unpaid checkouts can be cancelled without the confirmed-reservation cutoff, user-facing totals now prefer `totalPayment`, the payment sheet closes correctly even when the first query result is already terminal, and reserve/resume flows now keep time-sensitive validation and pending actions reachable.

### Reservation hardening
- **Slot-based reservation rules (`packages/backend/convex/reservations.ts`, `packages/backend/convex/schema.ts`, `packages/backend/convex/lib/reservation_utils.ts`):** reservations no longer depend on `tables.status === "available"` to allow future booking. The backend now validates booking windows server-side (future only, hourly slots, 10:00-21:00, max 7 days ahead), checks overlap with bounded indexed queries, and adds `reservations.by_startTime` plus `reservations.by_userId_startTime` to keep board/active-reservation lookups off full-table scans.
- **Checkout orchestration (`packages/backend/convex/payments.ts`, `packages/backend/convex/reservations.ts`):** customer checkout now starts through `payments.startReservationCheckout`, which creates the pending reservation, reuses an active pending payment link when possible, stores checkout metadata on the payment row, and compensates by cancelling the pending hold when Mayar link creation fails before a usable checkout session is recorded.
- **Operational table state (`packages/backend/convex/tables.ts`, `packages/backend/convex/reservations.ts`, `packages/backend/convex/orders.ts`):** table status is now treated as live floor state instead of a future-lock for all reservations. Public table queries derive `booked` from active confirmed reservations at the provided `referenceTimestamp`, `markOccupied` requires an active confirmed reservation, and `release` no longer hides an in-progress booking from the floor plan.
- **Customer/staff UX (`apps/web/src/components/reserve/reservation-form-sheet.tsx`, `apps/web/src/routes/_customer.reserve.tsx`, `apps/web/src/routes/_customer.my-reservations.tsx`, `apps/web/src/routes/staff.reservations.tsx`, `apps/web/src/routes/_public.events.$id.tsx`, `apps/web/src/routes/_public.table.$tableId.tsx`):** the reserve page now lets users inspect any active table and checks slot availability for the chosen schedule, pending reservations can resume payment from My Reservations, payment preflight surfaces missing phone numbers before retry, staff operations target the current/nearest relevant reservation, and event detail links now pass `eventId` plus a date prefill into `/reserve`.
- **Verification (`packages/backend/convex/reservations.test.ts`, `packages/backend/vitest.config.ts`, `packages/backend/package.json`):** added focused `convex-test` coverage for overlap rejection, booking-window validation, checkout rollback, idempotent payment confirmation, future-vs-current table status derivation, and release behavior during an active reservation. Frontend typecheck also uncovered and fixed the stale `user.image` access in `apps/web/src/components/user-menu.tsx`.

## 2026-04-17

### Post-launch review fixes — follow-up (#4, #6, #7)
- **Security — open-redirect on sign-in/sign-up (`apps/web/src/lib/safe-redirect.ts`, `sign-in-form.tsx`, `sign-up-form.tsx`):** new `safeRedirect()` helper applies a path-only allowlist to the `?redirect=` query param (must start with `/`, reject `//`, `/\`, absolute URLs, and scheme URIs like `javascript:`). Both forms now sanitize the value before `navigate(redirectTo)` and `authClient.signIn.social({ callbackURL: redirectTo })`, so a crafted link can no longer bounce freshly-authenticated users to an attacker-controlled origin.
- **Data integrity — user deletion cascade (`packages/backend/convex/auth.ts`):** the Better-Auth `onDelete` trigger no longer hard-deletes the `users` row (which would orphan `userId` references in reservations/orders/payments/notifications). Instead it **anonymizes** the record: clears `authId`, `avatarStorageId`, `avatarUrl`, `phone`; resets `name` → `"Pengguna dihapus"`, `email` → `deleted+<id>@cafe.local`, `role` → `"customer"`; deletes the avatar blob from Convex storage. Admin remains the sole trigger (via Convex / Better-Auth dashboard — no self-delete UI).
- **Analytics — `activeOrders` + 30-day `orderCount` (`packages/backend/convex/analytics.ts`):** `todayOverview.activeOrders` now sums live orders across `pending` + `preparing` + `ready` statuses via three parallel `by_status` index reads; `thirtyDayTrends[].orderCount` counts all orders whose `createdAt` falls inside each UTC day window (full scan for MVP, shares the `Promise.all` with the reservations query). `todayRevenue` intentionally stays reservation-only.
